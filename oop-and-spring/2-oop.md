# 객체 지향과 스프링

<br>

## 객체 지향

### 🔸 특징

##### 추상화
* 하위클래스에 존재하는 공통적인 메소드를 `인터페이스`로 정의
##### 캡슐화
* 속성과 기능을 멤버 변수와 멤버 함수로 만들어 `클래스`라는 캡슐에 넣음
* `정보 은닉`: 접근 지정자 `private`으로 멤버 변수 선언, 해당 변수에 접근하는 별도의 함수 정의
##### 상속
* 부모 클래스의 기능을 모두 또는 일부 수정하여 자식 클래스가 사용하는 것
* 코드 재사용
##### 다형성
* 같은 메서드가 각각의 객체에서 서로 다른 방법으로 응답
* 상속에서 효과를 발휘

### 🔸 객체 지향 프로그래밍
* 프로그램을 `객체`들의 `모임`으로 파악
* 객체들은 `메시지`를 주고받음 (`협력`)
* 프로그램이 `유연`하고 `변경`에 용이함

## 역할과 구현을 분리

>역할: 인터페이스 <br>
>구현: 인터페이스를 구현한 클래스, 객체


* 객체 설계 시, 역할(인터페이스)을 먼저 부여, 이 역할을 수행하는 구현 객체 만들기

* 클라이언트: 요청, 서버: 응답

* 클라이언트는 대상의 역할(인터페이스)만 알면 됨
* 클라이언트는 대상의 내부 구조를 몰라도 되고 구조가 변경되어도 상관 없음
* 클라이언트는 구현 대상 자체를 변경해도 영향을 받지 않음

* 클라이언트에 영향을 주지 않고 무한한 확장이 가능

* 한계
  * 인터페이스가 변하면 클라이언트, 서버 모두에 큰 변경이 발생
  
## 다형성
> 오버라이딩

인터페이스를 구현한 객체를 `실행 시점에 유연하게 변경` 가능

서비스: 클라이언트 Repository: 서버
클라이언트의 변경 없이 서버의 구현 기능을 유연하게 변경할 수 있음
## 객체 지향 설계의 5원칙: SOLID

### SRP (Single Responsibility Principle)
- 단일 책임 원칙

### DIP (Dependency Inversion Principle)
- 의존 관계 역전 원칙


```
다형성만으로는 OCP, DIP를 지킬 수 없음
```

## 객체 지향 설계

* 설계에서 `역할`과 `구현`을 분리
* 인터페이스를 도입하고 먼저 설계
  * 인터페이스의 도입은 추상화라는 비용을 발생시킴
  * 기능을 확장할 가능성이 없으면 구체 클래스를 직접사용, 향후 필요하면 리펙토링을 통해 인터페이스를 도입
